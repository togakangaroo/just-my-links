[AI]: # AWS Kendra

AWS Kendra is Amazon's enterprise search service that uses machine learning to enable natural language searching of content across your documents. It's designed specifically for document search with semantic understanding.

## Key Features
- Natural language processing capabilities
- Semantic search (understands meaning, not just keywords)
- Document ranking with relevance scoring
- Support for various document types (HTML, PDF, MS Office, etc.)
- Incremental learning from user interactions
- Built-in document connectors for popular repositories

## Technology
Kendra is a proprietary AWS service built on Amazon's own ML models, not an AWS wrapper around open source tools like Elasticsearch or Solr. It uses deep learning models specifically trained for document understanding and question answering.

## Pricing
Kendra has a tiered pricing model:

1. **Developer Edition**:
   - $0.75 per hour (~$550/month)
   - Up to 5 data sources
   - Maximum of 25,000 documents

2. **Enterprise Edition**:
   - $2.50 per hour (~$1,800/month)
   - Up to 50 data sources
   - Maximum of 100,000 documents

3. **Additional costs**:
   - Document storage: $0.05-$0.07 per 1,000 documents per month
   - Queries: $0.75-$1.00 per 1,000 queries

There is a **free tier** that includes:
- 750 hours of Developer Edition per month for the first 30 days
- Limited to a single index

For your personal use case with 10,000 documents, Kendra would work but might be relatively expensive compared to other options if you need it beyond the free trial period.

## AWS Kendra Pricing Model

- It's charged on an **hourly basis** for the entire time your index is provisioned, regardless of whether you're actively using it or not

## Embedding Generation in AWS Lambda

Yes, you can definitely generate embeddings for your documents in AWS Lambda, especially given their modest size:

- Short documents (1-10 pages) are ideal for embedding generation in Lambda
- Modern sentence/document embedding models like `all-MiniLM-L6-v2` are compact (~90MB) and efficient

The process would be:
1. Extract text from your documents (HTML parsing if needed)
2. Potentially chunk documents into paragraphs (optional for your document size)
3. Generate embeddings using a model like SentenceTransformers
4. Store results in SQLite

## SQLite Vector Search Capabilities

SQLite itself doesn't natively support vector search, but there are excellent extensions:

- **SQLite-VSS**: A purpose-built vector similarity search extension
- **sqlite-vss**: Supports both L2 and cosine distance metrics
- **Performance**: Works surprisingly well for collections up to millions of embeddings
- **Simplicity**: Far easier to manage than dedicated vector DBs for your scale

Example setup:
```sql
-- Load the extension
SELECT load_extension('sqlite-vss');

-- Create a virtual table with vector support
CREATE VIRTUAL TABLE embeddings USING vss0(
  embedding(384),  -- Dimension of your embeddings
  document_id UNINDEXED,
  content UNINDEXED
);
```

## SQLite Storage and Access Patterns

For your usage pattern (few queries per day), I recommend:

- **Storage**: Keep the SQLite DB in EFS (Elastic File System) mounted to your Lambda
  - EFS provides persistent storage that multiple Lambdas can access
  - Avoids loading the entire DB into memory each time
  - Handles concurrent access automatically

- **Alternative**: If you prefer S3:
  - Store the DB in S3
  - Download to Lambda's `/tmp` at the start of a request
  - Upload back to S3 after modifications
  - Use DynamoDB for a simple locking mechanism to prevent concurrent writes

- **Optimization**: Implement a warm-up mechanism to keep a Lambda instance alive between requests

Since your usage is personal and infrequent, this approach offers an excellent balance of simplicity, cost-effectiveness, and performance without over-engineering.

### 3. ChromaDB Implementation

```python
import chromadb
from chromadb.config import Settings

def handler(event, context):
    # Initialize ChromaDB with persistence to EFS
    client = chromadb.Client(Settings(
        chroma_db_impl="duckdb+parquet",
        persist_directory="/mnt/chromadb"  # EFS mount point
    ))

    # Get your collection
    collection = client.get_or_create_collection("documents")

    # For document indexing:
    if event.get('action') == 'index':
        document_content = event.get('content')
        document_id = event.get('id')
        metadata = event.get('metadata', {})

        collection.add(
            documents=[document_content],
            metadatas=[metadata],
            ids=[document_id]
        )
        return {"status": "indexed", "id": document_id}

    # For searching:
    elif event.get('action') == 'search':
        query = event.get('query')
        results = collection.query(
            query_texts=[query],
            n_results=10
        )
        return {
            "results": results
        }
```

## Advantages of This Approach

1. **Simplicity**:
   - ChromaDB handles all the vector search complexity
   - No need to manage SQLite extensions or implement custom vector search logic

2. **Persistence**:
   - EFS provides durable storage that persists between Lambda invocations
   - No need to load/save from S3 for each request

3. **Concurrency**:
   - ChromaDB handles concurrent access patterns
   - EFS supports multiple Lambda instances accessing the same data
   - No need to implement your own locking mechanism

4. **Performance**:
   - Once loaded, ChromaDB is optimized for vector similarity search
   - EFS provides reasonable read/write performance for your scale

## Considerations

- Keep your Lambda warm to avoid cold starts
- Monitor EFS usage costs (though they'll be minimal for your use case)
- Consider regular backups of your ChromaDB data to S3

This approach is significantly more straightforward than the SQLite solution while offering better semantic search capabilities out of the box. ChromaDB's files stored in EFS will persist between invocations, making this a clean, serverless solution for your document search system.
